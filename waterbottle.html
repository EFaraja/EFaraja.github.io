Project Overview

Simulating an Industry 4.0 Production Floor

The production line simulates the complete lifecycle of a water bottle across six manufacturing stages, from raw bottle supply through water filling, capping, branding, quality control, and final packaging.

The system continuously monitors production performance, equipment health, water tank levels, maintenance schedules, and defective products — surfacing all data through an interactive SCADA-style dashboard built entirely in Python.

The Challenge

Bridging software and the factory floor

Industrial production systems require reliable, low-latency communication between machines, human operators, and monitoring platforms. The challenge was to build a software architecture that faithfully simulates real factory operations while maintaining UI responsiveness, data integrity, and real-time visibility — entirely in Python.

Running production logic on the same thread as the GUI would cause the interface to freeze. Storing metrics in a flat file wouldn't support time-series analytics. A single-process deployment would fail to mirror the multi-service nature of real industrial stacks.

Key objectives

  • ⚡
    Real-time monitoring — sub-second metric updates without UI lag
  • 🔍
    Quality control simulation — automated defect detection per bottle
  • 🔧
    Maintenance tracking — CMMS with scheduled and reactive work orders
  • 💾
    Data persistence — time-series metrics stored in InfluxDB
  • 📊
    Analytics dashboards — Grafana for historical trend analysis
  • 🐳
    Containerized deployment — one-command startup via Docker Compose

System Flow

From a button press to a Grafana chart

Every component has a single job. Here's how they connect — from the entry point down to the database and back.

Module Structure 
launcher.py          ← entry point, starts everything
│
├── hmi.py           ← HMI window (operator controls production)
│   ├── production_line.py   ← the machine logic
│   │   ├── bottle.py        ← a single bottle object
│   │   ├── quality_control.py ← inspects each bottle
│   │   ├── cmms.py          ← tracks health, faults, work orders
│   │   └── database/influx_client.py ← writes to InfluxDB

└── cmms_dashboard.py  ← CMMS window (reads from InfluxDB)

01

You run launcher.py

It opens two separate windows using subprocess:

  • Window 1: hmi.py — the operator panel
  • Window 2: cmms_dashboard.py — the monitoring dashboard

They are two separate Python processes. InfluxDB is the bridge between them.

02

The HMI (hmi.py) controls everything

When you press START, this happens every second:

START button pressed → line.start() sets machine_state = "RUNNING" → auto_produce() loop begins → line.process_bottle() called every second

process_bottle() moves each bottle through 5 stages one step at a time:

Stage What happens
Bottle Supply bottle.has_bottle = True
Water Filling 95% chance bottle.filled = True
Capping 97% chance bottle.capped = True
Branding 96% chance bottle.branded = True
Quality Control QualityControl.inspect(bottle) — checks all 4 flags

At Quality Control, if any flag is False, the bottle is marked defective with a reason like "Missing Cap".

03

What happens after QC

✓ Bottle passes
  • good_bottles += 1
  • Data is pushed to InfluxDB
✗ Bottle fails
  • defective_bottles += 1
  • cmms.record_defect() — equipment health drops by 1 point
  • If 5+ defectives: cmms.report_fault() — creates a work order, logs the fault, stops the machine
04

Every bottle pushes two data packets to InfluxDB

# Measurement 1: production_line
bottles_produced, good_bottles, defective_bottles, efficiency_pct, machine_state (tag)
# Measurement 2: cmms_health
equipment_health, maintenance_status, work_orders_open, total_maintenance_actions
# Measurement 3: cmms_events (on fault / maintenance)
event_type (tag: "fault" or "maintenance"), message
05

The CMMS Dashboard reads from InfluxDB every 2 seconds

update_ui() runs on a 2-second timer. It calls InfluxReader which queries InfluxDB:

InfluxDB online?
YES
  • Read production_line + cmms_health + cmms_events
  • Show ● LIVE · InfluxDB in green
  • KPIs reflect HMI's real data
  • Status pill shows HMI's machine_state
NO
  • Fall back to its own offline simulator
  • Show SIMULATOR (InfluxDB offline) in amber
  • Each app runs independently

The CMMS also pushes station metrics (simulated wear per station) back to InfluxDB so Grafana can visualise them.

06

The CMMS (cmms.py) tracks machine health

The CMMS class is owned by ProductionLine inside the HMI process. It tracks:

AttributeWhat it means
equipment_healthStarts at 100, drops by 1 per defect
maintenance_statusNORMAL → WARNING → DUE
work_ordersList of fault tickets with OPEN / CLOSED status
fault_historyEvery fault ever recorded
maintenance_historyEvery maintenance action performed

When you press MAINT. in the HMI: perform_maintenance() resets health to 100, clears faults, schedules next maintenance in 30 days, and logs a maintenance event to InfluxDB.

07

Grafana connects directly to InfluxDB

Grafana reads the same measurements the HMI writes. Point Grafana at localhost:8086, org SRH, bucket production, and query any of:

production_linelive bottle counts and efficiency
cmms_healthequipment health trend over time
cmms_eventsfault and maintenance event timeline
station_metricsper-station wear and defect rates (written by CMMS dashboard)

CMMS — Computerised Maintenance Management

Keeping machines healthy, proactively

The CMMS module manages all maintenance activities for production equipment — from scheduled preventive maintenance triggered by run-hour thresholds to reactive work orders raised automatically when equipment health drops below defined limits. Every work order is tracked through a full lifecycle: open → assigned → in-progress → completed.

Data Collection & Monitoring

Every metric captured, nothing lost

Production metrics are written to InfluxDB every production cycle using the Python influxdb-client library. Measurements are tagged by equipment ID, production stage, and shift — making it easy to filter and aggregate data in Grafana for shift reports, equipment trending, and OEE analysis.

Multithreading & Communication

UI always responsive, production never blocked

Running production logic on the UI thread would freeze the interface. The architecture separates concerns across threads connected by a queue.Queue — the standard Python pattern for safe inter-thread communication without shared memory.

No UI Freezes

The production loop runs in a threading.Thread daemon — the interface stays smooth at all times.

Safe State Sharing

A threading.Lock protects shared production state; the queue delivers events without race conditions.

Clean Shutdown

A threading.Event signals the worker to finish its current cycle and exit gracefully on stop.

Docker Deployment

One command to start the entire stack

The monitoring infrastructure — InfluxDB and Grafana — is containerized via Docker Compose. Running docker compose up -d launches both services with pre-configured credentials, ports, persistent volumes, and an auto-provisioned Grafana data source pointing at InfluxDB.

docker-compose.yml

Conclusion

A foundation for real Industry 4.0 work

This project demonstrates how modern software engineering techniques — multithreading, containerization, time-series analytics, and event-driven design — can be applied directly to industrial automation challenges. Every architectural decision was made to mirror patterns used in real production environments, from the separation of UI and worker threads to the use of InfluxDB for high-frequency metric ingestion.

The system serves as a practical foundation for future extensions: OPC-UA integration, machine learning–based predictive maintenance, edge computing deployment, or a web-based HMI replacing the desktop client.
